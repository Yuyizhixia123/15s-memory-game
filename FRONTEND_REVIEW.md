# FRONTEND_REVIEW.md

本文档面向 Cursor / Claude Sonnet，用于后续前端美术与布局优化。请把本文档视为“当前 UI 状态审查 + 修改边界 + 可执行优化清单”，不是面向普通玩家的 README。

重要原则：当前项目已经有可玩的游戏闭环。后续前端优化应优先修视觉、布局、层级和可读性，不要顺手重写玩法系统。

## 1. 当前项目进展

### 1.1 项目是什么

《15秒记忆现场》是一个手机竖屏优先的网页小游戏原型。玩家观察 15 秒便利店监控画面，记住画面中发生的动态细节，然后回答一道关于刚才画面的四选一问题。

当前阶段是最小可运行、可试玩原型，不是商业完整产品。

### 1.2 当前核心玩法

1. 玩家点击“开始观察”。
2. 进入 15 秒观察阶段。
3. 便利店监控画面中按时间线播放若干事件。
4. 每个事件必须同步更新 `sceneState`、刷新 DOM 画面、写入 `eventLog`。
5. 观察结束后，`QuestionGenerator` 只基于本局 `eventLog` 生成一道问题。
6. 答对加 10 分，答错扣 1 条生命。
7. 生命归零进入结算页，并按完成轮数给出评级。

### 1.3 当前已完成的主要功能

- 首页、观察页、答题页、反馈页、结算页。
- 15 秒观察倒计时。
- 事件时间线播放。
- 每轮重置 `sceneState` 和 `eventLog`。
- 四选一问题生成和答题反馈。
- 分数、生命、轮数、结算评级。
- 重新开始。
- 快速点击防护，避免重复结算或重复跳轮。
- 右下角 Debug 面板。
- 便利店监控录像风格 UI。
- 分层 DOM 场景，不是一张整图。

### 1.4 当前已有系统

#### 事件系统

核心对象在 `script.js`：

- `DataConfig.difficultyTiers`：配置每个难度阶段的事件数量、允许事件类型、允许问题类型、观察时长和物品闪烁速度。
- `EventGenerator`：根据当前轮数生成事件列表，并用虚拟状态避免明显不合理事件。
- `TimelinePlayer`：按时间播放事件。事件处理函数负责更新 `sceneState`、调用 `SceneRenderer.render(sceneState)`、写入 `eventLog`。

当前支持事件包括：

- `customer_enter`
- `customer_exit`
- `item_taken`
- `shelf_item_disappear`
- `clock_change`
- `cashier_amount_change`
- `light_flash`
- `door_auto_open`
- `monitor_shadow`

注意：早期文档中提到的 `fridge_count_change` 和“冰柜最后剩几瓶饮料？”题目已被移除。当前冰柜中仍有“水”和“牛奶”商品，但不再有通用饮料库存事件或饮料数量题。

#### 问题系统

`QuestionGenerator` 只基于本局 `eventLog` 生成候选问题，然后随机选一道。当前题型包括：

- 最后进店的顾客穿什么颜色？
- 哪个商品被拿走了？
- 刚才灯闪了几次？
- 刚才门自动开合了几次？
- 最后店里有几名顾客？
- 哪个商品从货架消失了？
- 时钟最后显示几点？
- 收银台最后显示多少钱？
- 异常影子出现在哪里？
- 第一个进店的人是谁？
- 第一个发生变化的区域是什么？
- 最后发生的事件是什么类型？

#### `DifficultyManager`

`DifficultyManager` 根据轮数返回：

- `getEventCount(round)`
- `getAllowedEventTypes(round)`
- `getAllowedQuestionTypes(round)`
- `getObserveDuration(round)`
- `getItemFlashDuration(round)`

当前难度结构：

- 第 1-2 轮：3 个事件，简单事件为主。
- 第 3-5 轮：5 个事件，加入更多中等事件。
- 第 6 轮以后：7 个事件，加入异常事件和更细的问题类型。

#### Debug 面板

`DebugPanel` 默认隐藏，右下角可展开，显示：

- 当前阶段
- 当前轮数
- 当前分数
- 当前生命
- 当前 `sceneState`
- 当前 `eventLog`
- 当前 `currentQuestion`

#### 结算评级

`RatingManager` 根据成功完成轮数给出评级，例如：

- 刚摸到门
- 手感预热
- 节奏入门
- 反应在线
- 记忆高手
- 极限挑战者
- 人类高玩
- 非人类反应

#### 手机端 UI

项目以手机竖屏为主要目标，`.app-shell` 默认最大宽度约 430px，观察页顶部有状态栏，主画面为竖屏监控场景。当前也有少量 `@media` 规则分别处理窄屏和宽屏测试页面。

#### 便利店监控风格

当前视觉目标是轻微诡异的便利店监控录像：

- 深色背景。
- REC / CAM 标识。
- 监控时间戳。
- 扫描线。
- 轻微噪点。
- 暗角。
- 观察结束淡出变暗。
- 灯光闪烁。
- 异常黑影。

## 2. 当前前端视觉设计现状

### 2.1 整体视觉风格

整体是暗色、低饱和、带扫描线和噪点的监控录像风格。UI 有“便利店夜间监控”的方向，但仍是 DOM/CSS 原型级表现，不是最终美术。

优点：

- 已经明显区别于普通网页 UI。
- 监控 HUD、时间戳、REC、扫描线、暗角有效建立了主题。
- 观察结束后画面变暗，能防止玩家继续偷看。

不足：

- 便利店空间仍偏舞台化和拼贴感。
- 元素之间的透视、遮挡、比例和层级还不够统一。
- 商品标签、货架、人物、冰柜之间仍有“UI 组件拼在一起”的感觉。

### 2.2 当前便利店场景布局

场景使用五层 DOM：

- `backgroundLayer`：墙、地面、门、入口垫、货架、冰柜、收银台。
- `itemLayer`：水、牛奶、饭团、薯片、便当、糖果。
- `characterLayer`：门口通道、店员、顾客和离店动画顾客。
- `infoLayer`：时钟、收银金额、监控状态文字、门口招牌。
- `anomalyLayer`：灯光提示、异常黑影、红色噪点。

布局大致为：

- 上方：墙面、灯光提示、时钟、监控状态、门和招牌。
- 中部：两个货架。
- 左下：冷柜。
- 右下：收银台和店员。
- 右中：门口通道和顾客活动区。

### 2.3 人物、物品和环境表现方式

#### 人物

顾客和店员不再是简单圆圈，而是由头部、身体、手臂、腿部和标签组成的简化人物图形。顾客按颜色区分，店员在收银台附近。

现存问题：

- 人物比例与收银台、门、货架的比例仍不完全自然。
- 顾客进场动画较简单。
- 门口通道区域仍像 UI 框，不够像真实地面区域。

#### 物品

商品已经从“文字 x 数量”升级为多个简化图形：

- 水和牛奶：瓶状/盒状简化图形，放在冷柜区域。
- 饭团：三角饭团形状。
- 薯片：零食袋形状。
- 便当：盒状。
- 糖果：糖果形状。

当前数量显示方式：有多少个就渲染多少个图形，不再显示 `x2` 这类文本。

现存问题：

- 商品和货架层板的坐标仍较脆弱，容易在后续调整中错位。
- 商品名称标签目前在物品下方，可能与层板或其他商品接近。
- 商品本体和名称标签是同一组 DOM，后续高亮/移动时要避免标签参与动画。

#### 货架

货架有木质背景、三条层板、价签和“货架”标题。两个货架目前已对齐到同一高度。

现存问题：

- 货架仍偏大块矩形，真实便利店货架感不足。
- 价签和商品标签可能视觉竞争。
- 货架内部缺少更细的商品分区、挡板或陈列线。

#### 冰柜

冰柜有冷柜外框、玻璃质感和左右分割线。水和牛奶商品显示在冷柜内。

现存问题：

- 冰柜深度和内部层架不够明确。
- 冷柜内商品与玻璃门、层板的关系还可以更自然。
- 早期遗留的 `fridgeCount` 状态仍存在，但目前不再作为独立饮料数量玩法使用，后续不要误把它恢复成可见题目，除非明确要重做库存模型。

#### 收银台

收银台有柜台、收银机、扫码台、收银金额。金额显示已被放到收银台右半部分附近。

现存问题：

- 金额牌仍像浮层 UI，不完全像收银屏。
- 收银台与店员关系可更紧凑。

#### 背景

背景包含墙面、地板、门、入口垫、暗色监控纹理。海报和多余标语已被移除。

现存问题：

- 墙面、地板细节仍比较抽象。
- 便利店环境元素偏少，例如吊牌、货架背板、冷柜灯、过道阴影、门边框细节还可以加强。

### 2.4 手机端和桌面端布局情况

手机端：

- 默认最大宽度 430px，场景按竖屏设计。
- 已有 `@media (max-width: 360px)` 做窄屏微调。
- 画面可玩，但物品、标签、HUD、人物的空间较紧，后续修改容易造成重叠。

桌面端：

- `@media (min-width: 700px)` 会把 `.app-shell` 放宽到约 520px，并让场景更高。
- 适合浏览器测试，但桌面端仍是竖屏容器，不是横版布局。
- 当前桌面端更自然，但也可能暴露中上部留白或元素比例问题。

### 2.5 已完成的视觉成果

- 监控录像风格：REC、CAM、时间戳、扫描线、暗角、噪点。
- 便利店基本分区：门、货架、冰柜、收银台、门口通道。
- 商品已具备简化形状。
- 商品数量已显示为多个实际图形。
- 顾客和店员已从圆圈升级为简化人物。
- 顾客离店有向门移动并淡出的动画。
- 门自动开合已有动画。
- 物品高亮会按难度变化速度。
- 异常黑影已从单椭圆升级为简化剪影，并且问题答案与黑影位置对应。

## 3. 当前前端设计不足

### 3.1 仍不够像真实便利店

当前便利店识别度比早期好，但仍像“便利店元素拼贴图”。主要原因：

- 空间透视不统一。
- 货架、冰柜、收银台的比例不完全一致。
- 环境细节不足，缺少真实店内的连续结构，例如过道、墙角、冷柜灯带、货架背板、门外光源。

### 3.2 布局紧凑度仍需优化

中上部仍有较多功能性留白，而中下部元素密度较高。后续优化应在不遮挡问题相关元素的前提下，让墙面、门、货架和收银区之间更像一个连续空间。

### 3.3 物品和货架仍存在错位风险

当前商品坐标是写在 `DataConfig.items` 中的绝对像素位置，货架层板位置写在 CSS 中。这导致：

- 调整货架位置时容易忘记同步调整商品坐标。
- 商品名称标签改位置时容易造成穿模。
- 手机端和桌面端需要额外 `!important` 覆盖坐标。

这是当前前端最脆弱的视觉问题。

### 3.4 名称标签影响美观

商品名称标签有助于玩法识别，但会让画面更像 UI 标注，而不是自然监控画面。当前标签在物品下方，短期可读性较好，但会与货架层板、价签、商品本体争空间。

建议后续不要删除标签，除非同时重做更清晰的商品图形和问题文案。更可行的方向是弱化标签样式，让它像货架价签或商品贴纸。

### 3.5 环境细节不足

当前环境元素偏少，尤其是：

- 墙面缺少真实便利店信息层次。
- 地板格线比较抽象。
- 货架缺少背板和侧板。
- 冰柜缺少冷光、玻璃反光、层架关系。
- 灯光提示是显式文字 UI，不太像场景内灯具。
- 门口通道是虚线框，仍像调试区域。

### 3.6 手机端拥挤和可读性风险

手机端要同时容纳状态栏、监控 HUD、货架、冰柜、收银台、顾客、商品标签和异常效果。风险点：

- 商品标签过多时显得拥挤。
- 顾客进入后可能靠近门口通道边界。
- 收银金额、店员、收银台局部容易堆叠。
- 黑影和监控状态文字距离较近。

### 3.7 桌面端测试页面留白和比例问题

桌面端虽然更宽，但本质仍是竖屏容器。风险点：

- 场景中上部可能显得空。
- 门、货架、收银台比例可能在宽屏下显得分散。
- 需要通过媒体查询做桌面专用微调，而不是直接套手机坐标。

### 3.8 监控录像氛围还可以增强

已有扫描线和噪点，但还可以增强：

- 更微妙的色偏。
- 局部曝光不均。
- 监控镜头畸变感。
- 更像 CCTV 的 HUD 字体与边角标记。
- 事件发生时的短暂信号抖动。

注意：氛围增强不能影响玩家识别关键物品。

### 3.9 美术优化 vs 逻辑风险

纯美术优化：

- 颜色、阴影、边框、背景纹理。
- 商品图形内部细节。
- 货架、冰柜、门、收银台的 CSS 结构。
- HUD 装饰。
- 噪点、扫描线、暗角。

可能影响逻辑，必须谨慎：

- 改 `DataConfig.items` 的 `id`、`name`、`type`、`area`。
- 改事件类型字符串。
- 改 `eventLog` 记录字段。
- 改 `sceneState` 字段名或字段含义。
- 改 `QuestionGenerator` 出题依据。
- 改 `TimelinePlayer` 事件处理流程。
- 改 `DifficultyManager` 的事件数量、问题类型、轮次配置。

## 4. 给 Cursor / Claude Sonnet 的修改边界

### 4.1 允许修改

允许修改以下内容：

- `style.css`
- `index.html` 中展示层相关结构
- `script.js` 中少量与渲染表现相关的代码，例如：
  - `SceneRenderer` 内的 className
  - 纯视觉 DOM 结构
  - 视觉状态 class 切换
  - 不改变语义的坐标字段或表现映射

允许新增：

- CSS 变量。
- CSS 伪元素。
- 用于视觉结构的 DOM 子元素。
- 响应式媒体查询。
- 不参与游戏逻辑的纯展示类名。

### 4.2 禁止修改

禁止修改以下内容，除非用户明确要求并单独确认：

- `DifficultyManager` 核心逻辑。
- `EventGenerator` 核心逻辑。
- `QuestionGenerator` 核心逻辑。
- `eventLog` 数据结构。
- `sceneState` 数据结构。
- `currentQuestion` 数据结构。
- 分数、生命、轮数逻辑。
- 答题正确/错误判断逻辑。
- 游戏阶段切换逻辑。
- 观察时间、事件数量、问题类型配置。
- 事件类型字符串。
- 商品 `id` 字符串。
- 顾客 `id` 字符串。

### 4.3 修改后必须检查

每次前端优化后至少检查：

1. `node --check script.js`
2. 打开 `http://127.0.0.1:8000/`
3. 连续试玩 5 轮或做等价烟测
4. Debug 面板中 `eventLog` 仍有事件
5. 题目仍来自本局 `eventLog`
6. 手机窄屏和桌面宽屏都没有明显重叠

## 5. 前端美术优化建议

### P0：必须先修

#### P0-1 商品与货架坐标绑定过脆弱

问题：

商品位置在 `DataConfig.items` 中用绝对坐标写死，货架位置在 `style.css` 中写死。两者没有统一布局系统，导致商品容易穿模、悬空或与层板错位。

建议怎么改：

短期方案：建立统一注释和坐标规范，明确每一排货架层板的 y 值，并让商品 y 坐标与层板基准对应。

更好方案：在 `SceneRenderer.renderItems()` 中按 `shelfId` 和 `row` 计算视觉位置，而不是每个商品直接写绝对像素。但这会触及渲染映射，必须保持 `eventLog` 和商品 `id` 不变。

涉及文件：

- `script.js`：`DataConfig.items`、`SceneRenderer.renderItems()`
- `style.css`：`.store-shelf-*`、`.shelf-board`、`.scene-item`

核心逻辑风险：

中等。只改视觉坐标风险低；如果引入 `row/shelfId` 映射，要避免破坏商品 `id/name/type/count/area`。

验收标准：

- 所有货架商品本体都贴近层板。
- 标签不压住层板、不压住商品。
- 手机端和桌面端都没有明显错位。
- 物品被拿走或消失后，高亮仍对应正确商品。

#### P0-2 商品标签仍显得像 UI 浮层

问题：

商品名称标签虽然提升识别度，但与价签、货架标题争抢视觉层级，画面仍像标注图。

建议怎么改：

把商品标签弱化成货架价签风格：

- 更小、更统一。
- 颜色降低饱和度。
- 贴近货架层板边缘。
- 避免和 `SALE`、`¥4`、`¥5` 同时抢眼。

涉及文件：

- `style.css`：`.product-label`、`.shelf-price-tag`

核心逻辑风险：

低。

验收标准：

- 标签可读，但不成为画面最抢眼元素。
- 标签不遮挡商品本体。
- 标签不与货架价签重叠。

#### P0-3 手机端拥挤和点击/识别风险

问题：

窄屏下货架、商品、冰柜、收银台、顾客和 HUD 都在同一屏内，修改一点坐标就容易重叠。

建议怎么改：

优先维护 `@media (max-width: 360px)`：

- 为窄屏单独设置商品组宽度。
- 减小标签字号和 padding。
- 保证收银台金额、店员、顾客通道不压住。
- 必须实机或浏览器模拟窄屏检查。

涉及文件：

- `style.css`

核心逻辑风险：

低。

验收标准：

- 360px 宽度下能完整看清关键物品。
- 状态栏不遮挡场景。
- 问题相关物品变化能被玩家观察到。

### P1：强烈建议优化

#### P1-1 便利店空间缺少整体透视和层次

问题：

场景已具备门、货架、冰柜、收银台，但整体像平铺组件，空间感不够。

建议怎么改：

- 增加墙角线、地面分区、货架投影。
- 给门口、收银台、冰柜加入一致的光源方向。
- 加强地面网格透视，但不要旋转整体视角。

涉及文件：

- `style.css`
- 必要时 `SceneRenderer.renderBackground()` 增加纯展示 DOM。

核心逻辑风险：

低。

验收标准：

- 一眼看出是便利店内部。
- 元素之间像处在同一空间。
- 不影响商品识别。

#### P1-2 冰柜区域需要更像冰柜

问题：

冰柜已有玻璃与分割线，但内部层架、冷光、商品陈列关系仍可提升。

建议怎么改：

- 增加冷柜内部层板。
- 给水和牛奶加轻微玻璃遮罩。
- 增强冷光边缘。
- 让冰柜商品更像在柜内而不是贴在 UI 层上。

涉及文件：

- `style.css`
- `SceneRenderer.renderBackground()` 可加玻璃层 DOM，但不是必须。

核心逻辑风险：

低。

验收标准：

- 水和牛奶明确属于冷柜。
- 标签不遮挡瓶身。
- 冷柜本身不抢过商品变化高亮。

#### P1-3 监控异常黑影可以更自然

问题：

黑影已能按位置出现，但表现仍偏图标化。

建议怎么改：

- 增加短暂信号撕裂或画面色偏。
- 黑影出现时可有轻微位移或模糊脉冲。
- 位置仍必须与 `event.location` 一致。

涉及文件：

- `style.css`
- `SceneRenderer.getShadowLocationClass()` 只允许做视觉映射，不改题目逻辑。

核心逻辑风险：

中等。不能让视觉位置和 `eventLog.location` 脱节。

验收标准：

- 玩家能明显看到黑影。
- 左/中/右/下四个位置可区分。
- “异常影子出现在哪里？”答案和画面一致。

#### P1-4 收银金额应该更像收银屏

问题：

当前金额牌已靠近收银台右半，但仍像浮动 UI。

建议怎么改：

- 把金额显示整合进收银机屏幕。
- 保持 `cashier-amount` 可高亮。
- 不改变 `sceneState.cashierAmount`。

涉及文件：

- `style.css`
- `SceneRenderer.renderInfo()` 中金额节点 class 可微调。

核心逻辑风险：

低到中。不要删除金额 DOM，否则 `highlightedInfo === "cashier"` 的视觉反馈会失效。

验收标准：

- 金额像收银台的一部分。
- 收银金额变化时仍有明显高亮。
- 不遮挡店员。

### P2：后续美术增强

#### P2-1 增加轻微店内环境细节

问题：

当前环境偏简化，真实便利店氛围不足。

建议怎么改：

- 增加墙面小标签、冷柜灯带、货架侧边、地面污渍或反光。
- 细节应低对比，不能抢玩家注意力。

涉及文件：

- `style.css`
- 可选：`SceneRenderer.renderBackground()`

核心逻辑风险：

低。

验收标准：

- 场景更像便利店。
- 观察玩法不被装饰干扰。

#### P2-2 统一商品视觉语言

问题：

不同商品形状已有区分，但精细度、边框厚度、阴影方向还不完全统一。

建议怎么改：

- 统一商品边框粗细。
- 统一高光方向。
- 统一标签与商品的间距。
- 增加更具辨识度但低复杂度的 CSS 图形。

涉及文件：

- `style.css`

核心逻辑风险：

低。

验收标准：

- 商品不用文字也能大致识别。
- 多个商品放一起不显乱。

#### P2-3 增强监控录像质感

问题：

已有扫描线、噪点、暗角，但 CCTV 感还可增强。

建议怎么改：

- 加轻微镜头畸变边缘。
- 加偶发信号抖动 CSS 动画。
- 让 HUD 字体、颜色、边角标记更统一。

涉及文件：

- `style.css`

核心逻辑风险：

低。

验收标准：

- 监控感更强。
- 不降低商品、顾客、异常事件可读性。

#### P2-4 桌面端演示更完整

问题：

桌面端目前只是放宽竖屏容器，仍可能有留白或比例不自然。

建议怎么改：

- 保持竖屏游戏画布，但在宽屏下优化场景高度和元素间距。
- 不要改成横版游戏。
- 让桌面端更像一个完整手机预览。

涉及文件：

- `style.css`

核心逻辑风险：

低。

验收标准：

- 电脑浏览器打开时画面完整自然。
- 不影响手机端竖屏优先。

## 6. 推荐工作方式

建议 Cursor / Claude Sonnet 每次只做一个小目标，例如：

1. 只修商品与货架对齐。
2. 只优化冰柜视觉。
3. 只优化收银台金额。
4. 只增强监控 HUD。

每次改完都要说明：

- 改了哪些文件。
- 如何运行。
- 如何验证。
- 是否触及游戏逻辑。

不要一次性重写 `script.js`，也不要把场景改成图片。当前阶段仍应使用 DOM、CSS、文字标签、色块和简化图形。
